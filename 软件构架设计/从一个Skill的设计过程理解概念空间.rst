从一个Skill的设计过程理解概念空间
*********************************

介绍
====

本文记录我一个Skill的设计过程，这个过程也让我自己对概念空间的意义有了更深入的
理解，我把这个和AI的交互过程分享出来，以便说明概念空间的意义。

由于这里大部分都是和AI的交互过程，格式可能会比较混乱，我用如下格式区分每一个对
话：

.. note::

  <我的描述>

::

  <AI的答复>


和AI讨论模块设计Skill的过程
===========================

.. note::

   好，现在我们的设计已经可以接受了。我想你对照一下当前的skill module-design，
   告诉我如何改进这个定义，从而让AI可以一次写出当前这个设计文档这样的设计来。
   但不要直接修改当前的定义，给出建议就行了。

::

  看完了。对比当前 module-design skill 和 gqm_transport_design.md 的实际产出过程，以下是我的改进建议：

  ---
  1. 增加"层次归属分析"步骤

  当前 skill 直接从设计目的跳到设计逻辑，缺少一个关键问题：这个功能应该在哪一层实现？

  实际过程中，我们最初把长消息管理（状态机、CopyIn/Out、msg_type）全放在 redisGqm 层，经过反复讨论才意识到这些应该推到 libgqm 层。如果 skill
  在设计逻辑之前要求分析"功能与已有模块的归属关系"，可以一次到位：

  ▎ 在建立设计逻辑前，先分析目标功能与系统已有模块的关系。对于每个子功能，判断它应该在本模块实现，还是属于已有依赖模块的能力范围。如果属于依赖
  ▎ 模块，考虑是否应该先增强依赖模块，而不是在本层重复实现。

  2. 增加"封装策略"的显式决策

  当前 skill 没有讨论模块与其依赖之间的边界策略。实际设计中 redisGqm 做了一个关键决策：薄封装 + 直接暴露下层接口（直接暴露
  qm_u_rpc_local_desc，不重新包装）。

  ▎ 在描述设计逻辑时，明确模块与依赖层的封装策略：是完全封装下层接口、薄封装并暴露部分下层能力、还是直接暴露。这个决策决定了接口定义的范围和复
  ▎ 杂度。

  3. 将"一致性校验"细化为结构化自检清单

  当前 skill 的"一致性校验"只是一个笼统的"检查有没有逻辑冲突"。实际过程中，我们通过 review
  发现了大量问题（状态机笔误、命名不一致、接口签名缺失、非法状态转换遗漏）。建议把校验变成具体的 checklist：

  ▎ 一致性校验应包含以下维度的逐项检查：
  ▎ - 概念一致性：同一概念是否全文使用相同术语，没有同义异名或异义同名
  ▎ - 状态完备性：每个状态机的每个状态是否都处理了所有可能的输入事件（包括非法事件的处理策略）
  ▎ - 接口完备性：设计逻辑中提到的每个操作是否都有对应的接口
  ▎ - 层次一致性：本层接口是否承担了属于其他层的职责

  4. 增加"错误处理策略"作为设计逻辑的一部分

  实际过程中，状态机的错误处理策略经过了几轮重构（逐状态处理 → 统一 ERROR → 分离正常/错误路径）。如果 skill
  在设计逻辑阶段就要求明确错误策略，可以减少迭代：

  ▎ 在设计逻辑中，对可能出错的部分明确错误处理策略：是逐场景分别处理，还是统一恢复机制？错误处理是在本层还是委托给调用者？这个策略应在设计逻辑
  ▎ 中早期确定，因为它影响接口的返回值语义和状态机建模。

  5. 补充接口设计的"返回值语义"指导

  实际过程中，CopyIn/Out 的返回值语义经过多轮讨论才收敛（返回 copied bytes → 返回状态码 + copied 输出参数）。当前 skill
  只说了"重点放在功能上"，缺少对返回值设计的指导：

  ▎ 接口定义中，如果函数可能失败，明确返回值策略：是用返回值表示错误码（输出参数传结果），还是用返回值传数据（特殊值表示错误）。这应与错误处理
  ▎ 策略保持一致。

  6. 增加"设计验证"步骤

  实际过程中，我们通过写 libgqm 的代码和单元测试验证了状态机的正确性（发现了一个 copy_out 的状态校验 bug）。当前 skill
  的"概念验证"只提了一句，没有强调用实现来验证设计：

  ▎ 对设计中的关键逻辑（尤其是状态机、协议、并发逻辑），建议通过编写最小实现或单元测试来验证设计的正确性，而不是仅靠推理。实现验证后再回到设计
  ▎ 文档修正发现的问题。

  ---
  以上是让 AI 能一次产出接近最终质量的设计文档的核心改进。总结起来就是：在设计前多问"该不该做"，在设计后用 checklist
  自检，用实现验证关键逻辑。

.. note::

  你这个建议太关注这“一次”设计的经验了，可能过度要求，导致在其他设计中效果不佳。
  我认为我更关注AI能清晰表达：

  1. 目标。这样我可以确认一下它对“我们要做什么”的理解和我是否一致。
  2. 设计逻辑。我可以确认它关注的“主要矛盾和矛盾的主要方面”是否和我一致，它的逻辑是否严密。
  3. 核心数据结构。数据结构控制所有的API能实现成什么样，核心数据结构容纳不了的东西，API什么设计都没法提供新功能了。
  4. 最终的用户接口。这样我可以详细看它的设计结果，是否和我一样。

  在这个基础之上，其他都是锦上添花，用来保证这些部分写得更有针对性一些而已。你
  觉得我这样考虑是否正确？据此，你认为我还可以如何改进这个skill？

::

  你这个考虑是正确的。这四个要素形成一个严格的依赖链：目标决定逻辑，逻辑决定数据结构，数据结构决定接口。任何一层出错，后面的就全错了。所以
  skill 的核心职责就是引导 AI 把这四层一步步想清楚，而不是堆砌模板和技巧。

  基于这个原则，我认为当前 skill 有几个可以改进的地方：

  1. 设计逻辑缺少"找矛盾"的引导

  当前 skill 对设计逻辑的描述偏重于"用什么方法表达"（逻辑树、交互图、状态机等），但没说清楚核心要做什么。实际上设计逻辑的核心是找到主要矛盾并
  给出解决方案。建议补充：

  ▎ 设计逻辑的核心是识别实现目标过程中的关键挑战和取舍。不要罗列所有细节，而是抓住2-3个核心问题，说明你选择了什么方案、为什么不用其他方案。如
  ▎ 果某个问题没有取舍（只有一种合理方案），简述结论即可，不需要展开。

  2. "概念空间"不应独立成章

  当前 skill 把"概念空间"作为一个独立的关注点，用了很多篇幅解释。但正如你自己说的，概念是"在设计逻辑中产生的"。把它独立出来容易导致 AI
  为了定义概念而定义概念。建议把概念空间的引导合并到设计逻辑中，作为写设计逻辑时的注意事项：

  ▎ 在描述设计逻辑时，自然地定义你引入的新概念。确保全文对这些概念的使用一致，不要在前面用"长消息"、后面又用"超长数据"指代同一个东西。

  3. 核心数据结构缺少"与设计逻辑的对应"

  当前 skill 说"核心数据结构"是围绕它提供功能的，这没错，但没有强调一个关键点：数据结构必须能承载设计逻辑中承诺的所有能力。建议补充：

  ▎ 写完核心数据结构后，回头检查设计逻辑中提到的每个能力，是否都能通过这些数据结构的成员实现。如果某个能力找不到对应的数据支撑，说明数据结构定
  ▎ 义有遗漏，或者该能力本身不成立。

  4. 接口定义缺少"与目标的追溯"

  当前 skill 说接口要"对应设计逻辑和概念空间"，这还不够。接口最终要追溯到设计目的——每个接口的存在都必须是为了达成某个设计目标。建议补充：

  ▎ 列出接口后，检查每个接口服务于哪个设计目标。如果一个接口不对应任何明确的目标，考虑它是否必要。同样，检查每个设计目标是否都有对应的接口来达
  ▎ 成。

  5. 一致性校验应该围绕四个要素的依赖链

  当前的一致性校验是泛泛的"检查逻辑冲突"。基于你的四要素框架，校验应该变成逐层检查这条依赖链：

  ▎ 一致性校验沿着依赖链逐层检查：
  ▎ - 逻辑 → 目标：设计逻辑是否覆盖了所有目标？
  ▎ - 数据结构 → 逻辑：数据结构是否支撑了所有设计逻辑中的能力？
  ▎ - 接口 → 目标：每个目标是否都有对应的接口？每个接口是否都服务于某个目标？
  ▎ - 全文术语：同一概念是否全文使用相同名称？

  6. 设计目的中关于版本/时间的描述可以简化

  当前 skill
  花了很多篇幅讲"要标注设计基于哪个版本"。这没错，但不是核心关注点。建议简化为一句话要求，把更多篇幅留给"帮助用户确认理解是否一致"：

  ▎ 简要标注设计基于的项目状态（git revision 或日期），然后用自己的话总结用户的设计期望。总结的目的是让用户确认你的理解和他的意图一致。

  ---
  总结一下改进方向：当前 skill 的结构是对的（目标→逻辑→数据结构→接口），但内部引导偏重于"怎么表达"而非"怎么思考"。改进的核心是让每一步都围绕依
  赖链来思考——这一层是否支撑了上一层，是否被下一层支撑。其他技巧（层次分析、错误策略、状态机验证）都是为这条链服务的具体手段，不应喧宾夺主。

.. note::

  我觉得你对于概念空间定义的顾虑是有道理的，但我独立写这个部分也有我的考量：对
  我来说，每个设计从目标到核心数据结构和接口的过程，就是一个概念空间定义的过程，
  因为在设计之初，我们没有新的概念，我们都是用已有的概念来描述我们要做的那个东
  西的，我们进行设计的过程就是重构当前的概念，比如我们原来只有gqm消息的概念，
  只有发一般消息的概念，但我们进行设计后，我们说，我们还可以发gqm的消息，但我
  们为了发一个很长的RPC，我们可以组合多个gqm消息，从而形成一个“超长消息发送”的
  概念，最终我们的接口定义上就出现qm_u_long_msg_xxx()这样的表述了，所以，我必
  须让AI保证在这个过程中给我定义出这些新概念来，这样我就能理解那些新的接口了。
  否则它就有可能直接用以前的概念来给我解释这些新的接口，这是设计中和人交流最重
  要的一步，你觉得我可以怎么把这一点说清楚给AI听？

::

  你说得对，我之前理解浅了。你说的不是"设计逻辑中顺带提一下概念"，而是设计本身就是创造新概念的过程——从已有的"GQM消息"概念中诞生出"超长消息"这
  个新概念，然后接口才能叫 qm_u_long_msg_xxx()。如果 AI
  没有做这个概念创造，它就会用"多次GQM消息加MORE标记"这种旧概念的组合来解释接口，读者就看不到抽象。

  我认为把这一点说清楚给 AI 听，关键是说三件事：

  1. 概念创造是设计的本质工作，不是副产品

  ▎ 设计过程就是从已有概念中构造新概念的过程。你在设计逻辑中每解决一个核心问题，都应该检查：这个问题是否需要一个新概念来描述它的解决方案？如果
  ▎ 需要，显式地给它命名，说明它由哪些已有概念构成、它封装了什么复杂性。不要只用已有概念的组合来描述解决方案而不给它命名。

  2. 新概念一旦定义，后续必须用它而不是退回旧概念

  ▎ 定义了新概念后，核心数据结构和接口定义中必须直接使用这些新概念名称。比如你定义了"超长消息"这个概念，接口就应该叫 send_long_msg()，而不是
  ▎ send_multiple_msgs_with_more_flag()。如果发现某个接口不得不用旧概念来解释，说明缺少了某个概念定义。

  3. 把这放在设计逻辑的开头，作为思考方式

  不需要独立章节，而是在设计逻辑的引导中加入这段话：

  ▎ 开始写设计逻辑之前，先想清楚：为了达成设计目标，你需要引入哪些当前系统中不存在的新概念？这些概念是从已有概念的什么组合中产生的？它们封装了
  ▎ 什么复杂性？在设计逻辑中，先定义这些概念，然后用它们来描述你的方案。

  我觉得这样的表述既保留了概念空间的重要性，又不把它变成一个独立的形式化步骤——它就是设计逻辑思考方式的一部分。你觉得这个方向对吗？

总结
====

我想表达的内容在上面的交互过程中我已经表达了，就概念空间的地位这件事本身，我也
没有什么需要补充了。

我之前写了一篇\ :doc:`AI编程中人的作用`\ 。有同事看了以后问我，既然AI做不到对
新事物的本质的抽象，那么我们有什么高效高质量的方法，通过结构化的描述让AI给我们
写出符合期望的输出？

我想用这个例子来解释我的观点：就拿我和AI协同做的这个设计来说，整个设计过程我重
构了六、七次，每次都在打破我自己原来给AI的要求，AI自己根本就想不到我想到的那些
东西，比如某个层薄了，我会考虑压缩或者合并它，某个状态机复杂了，我会想到封装成
模块，这些，几乎都不是逻辑判断，而是“隐层判断”，我是凭直觉做的，不是综合了几十
个要素，最终“推理”出这个地方就是要合并，就是要分解。这些设计，每个都是“无中生
有”，没有任何道理，就是看到这种情况我就蹦出来了，我不能给定一组条件，说：你每
次遇到这个组合就这样。但我的设计明显就是比它好，因为做到最后，我的代码量，可调
试性，可测试性，都比AI按它“直接的理解”好。

但很重要的一点是，我自己也必须做六七次迭代我才有现在的结果的。我也不是一次就能
成型的。我是看到一个样子，决定可以做什么调整，但调整以后，发现另一个机会点，然
后我做一轮修改，慢慢逼近一个最优的结果的。

这个过程，我一直需要AI的协助，它能很快给我做一轮细节设计甚至编码。而且上面的交
互过程，你可以看到其实AI很多时候表达能力是比我强的，我给它下的要求，很多它的描
述是比我好的，比如这段：

  | 设计本身就是创造新概念的过程——从已有的"GQM消息"概念中诞生出"超长消息"这
  | 个新概念，然后接口才能叫 qm_u_long_msg_xxx()。如果 AI没有做这个概念创造，
  | 它就会用"多次GQM消息加MORE标记"这种旧概念的组合来解释接口，读者就看不到抽象。

我觉得我说了半天白说了，它这么短一段就说清楚了。

所以我和它的互相辅助过程，乃至我自己和自己的交互过程，都是交织在一起的，这就是
“具体问题具体分析”的工作方法，我们很多人期望的“定义一个结构化方法就完成交互”，
这有点知识分子的傲慢了。这是对待机械智能的态度（逻辑的简单重复），但高级智能不是
这样工作的，我们老想着用这种方法对待AI甚至自己，最终只会让自己离智能越来越远，
退化成一台机器，变成“遇到A情况就做这个，遇到B情况就做那个”，完全放弃了自己的隐
层上的归纳和总结能力。

补充：什么叫“隐层判断”
======================

这个小节根据和一些同事的讨论补充解释一下什么叫“隐层判断”。

人工神经网络主要就是在做矩阵乘法，矩阵乘法的本质就是在做线性代数的变维。简单比
喻就是：我们用一组数字组成的向量去代表一个对象，比如一个人，我们用[年龄，身高，
性别，体重，……]去表示他，这每个数字就是一个维度，一个这样的向量就叫一个
embedding，中文翻译为“嵌入”，表示嵌在这个人内部的某种“固有属性”。

而变维是什么意思呢？就是我们研究规律的时候，换一个角度来看这个人，比如我们基于
他的年龄身高性别数据，我们估计它另一组维度：[科学家特质，运动员特质，学霸特质，
学渣特质……]，这也是embedding，也代表那个人的内在固有属性，但我们换了一个维度了，
这就是矩阵乘法干的事情。我们从一个维度变维到另一个维度，从另一个角度发现人的规
律。

但我这里是为了让你看明白，才用科学家特质，运动员特质来形容这些变维后的
embedding，实际上人工神经网络里面的维度常常没有“意思”，我们只是做个变维，尝试
发现规律，它的维度是被某种内在的规律驱动的，我们没法说它是什么，因为那个东西没
有我们定义过的概念和它对应。我们所有能说出来的，都是Token。这就是为什么《道德
经》里面一开始就强调“道可道，非常道”，我们能想的，能说的，都是Token，但世界的
规律和信息是Embedding，甚至不是Embedding，因为那个被描述的对象也被重新分类了，
我们没有Token可以和它对应，所以我们没法说。所以，在神经网络两头的接口，我们就
叫“显层”，这些地方参与计算的Embedding，我们基本是知道每个维度表示什么的，但到
了网络内部，我们没法说是什么意思，这个部分就叫“隐层”，这个部分我们没有建立逻辑，
因为逻辑只能用Token表达。

我们能和AI交流，能说的，一定是Token，但我们的判断力并不只在Token上，我们的判断
力，大部分在隐层的权重上。人现在很大程度上比AI强，是人看到大量的数据，能总结出
我们独特的Token，有这个Token以后我们才能谈逻辑。

举个例子，面对某个场景，你看到4个人气势汹汹地提着斧头向你扑来，你应该怎么办？
AI说：快跑。但你可能做出不同的判断：没事，领头的是我爸，他不是来砍我的，而是来
保护我的。你注意到了“领头的是我爸”这个特征，这是你从无数信息中提取出来的，这是
逻辑给不了你的，因为Token处理不了特别多的信息，大量的信息只能通过隐层综合来发
现。这种方法不可靠（比如可能你忘了，你昨晚偷了家里的钱去打赏主播了，你爸来砍的
就是你），但它处理效率高啊。

AI也有隐层，你也有隐层，但大家的内部维度不同，训练数据不同，总结能力，产生
Token的能力也不同。现在的情况就是大部分时候，人的Token能力总体上是普遍高于AI的
（比如我上面呈现的例子，这种情况，几乎每个靠谱的程序员都能体会到），当然不排除
不少人会差到不如AI，但无论如何，你不应该不断训练自己，不用你的隐层判断能力，把
自己训练成一台只能根据固定的套路，遇到一种情况就应激一种反应的机器。
