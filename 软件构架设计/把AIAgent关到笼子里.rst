把AIAgent关到笼子里的一点体验总结
*********************************

介绍
====

最近一直用一般用户直接运行AI Agent编程，有时甚至直接用Agent直接解决本地不少配
置和安装问题，感觉效率提升的同时，也常常有点后怕：为了让AI好用，基本上工具会越
给越多，而且基本上没空检查它要运行的每个脚本，这东西任意读机器上的配置，甚至执
行sudo，这些信息几乎都会被Agent发到LLM服务器上，这里面什么都有：主机名，保存在
各种配置中的密码，apikey，ip地址，mac地址……这些东西十有八九会被运营商拿来训练
新的模型。就算这些运行商脱敏做得好，但某些pattern的组合很难说不会被LLM永久记住，
谁知道那天不会变成公开给全世界的某种“例子”，或者“知识”呢？

所以，我还是想着尽快把这个东西装到笼子中，不能继续这样裸奔了。本文是做这件事的
一些体验总结。

对了，我用的主力机器是Debian 13。

方案选择
========

做这件事，我第一反应是三个选择：

* 加一个用户
* 加一个容器
* 加一个虚拟机

一个比一个安全，但同时一个比一个慢。我先试了第一个方案，各方面都可以接受，但最
大的问题是：我给不给sudo权限？不给不少事情干不了，给了我自己不安全了。所以这个
让它搞用户态开发是可以的，像我这样经常让它干危险动作的，好像就不合适了。

这个方案我到最后是创建了几个dev1, dev2这样的账户，不给sudo权限，先登录到这种用
户，做一般的开发，如果需要使用高级功能，再su到有sudo权限的主用户上（比如叫
kenneth），在独立的shell中对数据进行备份，或者执行系统操作。

而且我觉得这样弄我还是不大放心，因为我的/etc目录对devN还是暴露的，我也无法肯定
这里有没有我想不到的风险。这种方法很难当作主方案来用。

然后我选择第二个方案，现在Docker在国内几乎就没法用，所以我选择直接用LXC。LXC挺
不成熟的，但胜在算是Linux的原生方案，死也死不到哪里去。它的镜像服务器是社区维
护的，基本的社区发行版都有。用用户态权限创建镜像后（lxc-create），以一个映射
user id的形式存在，比如host的用户kenneth是1000，可以用/etc/subuid和subgid映射
到100000，那么创建的容器的0用户就会映射到100000，而容器的1000用户会映射到
101000，这样容器基本是被封住的。其他文件都是~/.local/share/lxc里面分容器的
rootfs中，被隔离开了。其他目录基本上都可以用bind mount映射过去。这样，我主登录
的用户还是kenneth，一切原来的功能都可以，开发则隔离在100000-165535这个用户区间
内，甚至1000和100000互相也是隔离的，在1000这个用户下需要用lxc-usernsexec才能切
换成100000系列用户来执行。

这样看起来基本上大部分系统级的功能都可以干了，只是影响不了真正的本地系统，如果
我需要控制本地系统，看容器里面怎么干的，在外面人工做一次就是了。

网络是通过veth连外部的网桥的（通过/etc/lxc/lxc-usernet配置，配置很简单），可以
直接用nat连公网，也可以直接连host的主机，只要别把ssh key给容器，它就干涉不了外
面，ufw等防火墙全部可以在host中完成拦截。

这个方案暂时看来是最舒服的，基本上我创建一个基本系统，在clone一个加了很多基本
配置的富功能系统，然后再clone几个dev系统，就可以随便玩了。现在Debian 13的
lxc-copy有bug，但容器文件只是kenneth和100000这两种用户的，用一般用户权限就可以
完成全部拷贝，手工拷一次就是了。

最后一个验证的是虚拟机方案，这个没有什么特别的经验要说的，我用的是VMM+KVM的方
案，基本上没有遇到什么问题，就是吃资源比较多，肯定没法像容器那样开很多实例，映
射设备过去也比较不稳定（比如把host连的手机暴露给虚拟机的adb，这经常会断），其
他的体验基本上和容器一样，就是慢。

总结
====

这个验证我其实还没有做完，但因为现在够用了，我暂时就试到这里，其他经验以后再做
什么验证我再补充。

我其他的感想主要是针对之前讨论的一个问题的：到底Agent OS的需求是什么，我现在觉
得主要是Container，我觉得Agent OS应该集成足够好用的Container的技术，随时可以开
很多的Container，可以很容易设置限制范围，可以很容易把本地的资源根据需要映射给
容器，这样在未来才能保证我们大量引入Agent，而不会担心把自己的信息直接暴露到整
个网络上。
